🔴 G-C1 — GraphPipeline.run() raises AttributeError on every entity graph

narrative_metrics: Optional[Dict[str, float]] = None
Helper:
def _to_graph_structure(g: Optional[Dict[str, Dict[str, float]]]) -> Optional[GraphStructure]:
    if not g:
        return None
    nodes = sorted(g.keys() | {n for nbrs in g.values() for n in nbrs})
    edges = {n: sorted(g.get(n, {}).keys()) for n in nodes}
    return GraphStructure(nodes=nodes, edges=edges)

🔴 G-C5 — entity_graph and narrative_graph types disagree across the codebase
•	File: All of src/graph/
•	Issue: EntityGraphBuilder.build_graph and NarrativeGraphBuilder.build_graph return weighted Dict[str, Dict[str, float]]. But GraphAnalyzer.analyze, GraphExplainer._node_importance/_edge_importance, GraphEmbeddingGenerator._to_nx, and GraphFeatureExtractor.extract_from_graphs all annotate the input as Dict[str, List[str]]. The code "works" because Python iteration over a dict yields its keys, but:
o	GraphAnalyzer.normalize_graph then calls for nbr in neighbors: if isinstance(nbr, str) — drops the weights entirely. Edge weights are lost between builder → analyzer.
o	GraphExplainer._edge_importance does edge_scores[edge] = edge_scores.get(edge,0)+1.0 — counts every edge as 1.0 instead of using the actual weight.
o	_node_importance does degree = len(neighbors) — fine for unweighted, but ignores edge strength.
•	Why broken: The whole "🔥 weighted co-occurrence" claim in the builders is silently undone by every downstream consumer. Heavy edges and one-off mentions count the same.
•	Fix: Pick one canonical type. Recommend Dict[str, Dict[str, float]] everywhere (it's a strict superset):
# graph_analysis.py — normalize_graph
for nbr, w in (neighbors.items() if isinstance(neighbors, dict) else ((n, 1.0) for n in neighbors)):
    ...
Then use the weights when computing degree = sum(weights), edge_score = w, etc. Also update type hints across graph_analysis.py, graph_explainer.py, graph_features.py, graph_embeddings.py, graph_visualization.py.

🟠 2. PERFORMANCE BOTTLENECKS
🟠 G-P1 — Triple-build of the same graph per request
•	Location: GraphPipeline.run builds the entity graph once at L145, then GraphFeatureExtractor.extract_from_graphs (called at L177) re-builds it implicitly via self.entity_builder.extract_features(entity_graph) (OK, shared input), but extract_from_graphs also runs self.analyzer.analyze(entity_graph) (L115) and graph_embedding_vector(entity_graph) (L122). Each of those internally normalises and undirected-ifies the graph independently. Net cost: the same O(N+E) graph is normalised 3× per request.
•	Fix: Compute normalised = to_undirected(normalize_graph(g)) ONCE in the pipeline and pass it down (or memoize on the dict id):
# src/graph/graph_pipeline.py — after building entity_graph
if entity_graph:
    entity_graph = to_undirected(normalize_graph(entity_graph))   # canonicalise once
Then have GraphAnalyzer/graph_embedding_vector accept an already_canonical=True flag to skip the re-pass.
•	Expected speedup: ~2.5× on the graph stage (currently dominated by Python-loop normalisation).
🟠 G-P2 — O(N²) Python loops in extract_features, compute_graph_metrics, _node2vec
•	Location: entity_graph.py:174–180, graph_analysis.py:103–107, graph_embeddings.py:142–177
•	Issue: Triangle counting and clustering are written as nested Python loops over node neighbours; _node2vec is a Python loop with np.random.choice per step (the worst case for numpy overhead). On a 100-node graph that's ~10⁴ Python-level iterations per request.
•	Fix: Replace with sparse-matrix ops:
import scipy.sparse as sp
A = sp.csr_matrix(adj)             # build once from adjacency
triangles = (A @ A @ A).diagonal().sum() / 6
triplets = degrees * (degrees - 1) / 2
clustering = (3 * triangles) / triplets.sum()
For node2vec, use scipy.sparse.csgraph random walks or replace with a single eigen-decomp (you already have spectral_eigen_embedding).
•	Expected speedup: 10–50× on graphs with >50 nodes.
🟠 G-P3 — One-graph-at-a-time pipeline; no batching
•	Location: every GraphPipeline.run(text) callsite
•	Issue: GraphPipeline.run accepts a single text and the spaCy parser runs on it alone. EntityGraphBuilder.__init__ instantiates a new GraphPipeline (and therefore a fresh EntityGraphBuilder and NarrativeGraphBuilder) on every call to FeaturePreparer(...) because each component instantiates its own. With n_process=1 and no nlp.pipe, batch inference parses N texts serially.
•	Fix: Add GraphPipeline.run_batch(texts: List[str]) that uses nlp.pipe(texts, batch_size=32, n_process=1) for the entity graph and vectorises the narrative-graph regex extraction across documents:
def run_batch(self, texts: list[str]) -> list[dict]:
    docs = list(self.entity_graph_builder.nlp.pipe(texts, batch_size=32))
    return [self._run_from_doc(text, doc) for text, doc in zip(texts, docs)]
•	Expected speedup: 3–5× on batch inference paths (batch_inference.py, feature_pipeline.batch_extract).
🟠 G-P4 — Eigen decomposition on dense nx.to_numpy_array
•	Location: graph_embeddings.py:229, 239
•	Issue: nx.to_numpy_array(G) builds an O(N²) dense matrix even for sparse graphs; np.linalg.eigvalsh is O(N³). For long articles (~200 nodes) that's 8M flops per request.
•	Fix: Use sparse Laplacian + scipy.sparse.linalg.eigsh(k=spectral_dim, which='LA') — converges in O(k·N) for sparse adjacency.

🟡 3. SEMANTIC ISSUES
🟡 G-S1 — Narrative "graph" is actually a sentence-keyword bigram chain, not a graph
•	File: src/graph/narrative_graph_builder.py:104–123
•	Issue: For each sentence, _extract_keywords returns the most-frequent-tokens-by-count, then edges are added between every keyword in sentence i and every keyword in sentence i+1. There is no co-reference, no syntactic dependency, no entity linking. So the "narrative graph" is just top-k tokens per sentence × top-k tokens per next sentence. Calling narrative_graph_centralization on this is mostly noise.
•	Fix (semantic): Tie the narrative graph to the spaCy noun-chunks / entities you've already extracted in EntityGraphBuilder, and edge nodes by shared sentence membership (giving genuine co-occurrence) plus temporal succession between sentences. Drop the regex tokeniser.
🟡 G-S2 — Edge in entity graph is not directional but stored as if it were
•	File: src/graph/entity_graph.py:131–135
•	Issue: graph[a][b] += 1.0; graph[b][a] += 1.0 then calls to_undirected(graph) which sums them again — so each undirected co-occurrence ends up with weight 4 * count instead of 2 * count. The relative ordering survives, but downstream entropy / centralisation / density calculations all see inflated weights.
•	Fix: Either store one direction at build time and let to_undirected symmetrize, or skip the to_undirected call (it's already symmetric by construction):
graph[a][b] += 1.0      # only one direction at build time
graph[b][a] += 1.0      # OR drop one and call to_undirected once
# then DROP graph = to_undirected(graph) at L135

🚀 5. GPU / MEMORY ISSUES
🚀 G-G1 — All graph features are CPU numpy.float32 → forced CPU↔GPU copy at the model boundary
•	File: src/graph/graph_features.py:207, src/graph/graph_embeddings.py (returns np.ndarray)
•	Issue: The graph layer is intentionally CPU-only (good for the spaCy stage), but the resulting feature vector is concatenated as np.float32 and only copied to GPU inside HybridTruthLensModel.forward. With batched inference this is N separate host→device syncs. The extract_feature_vector_from_features doesn't know the target device.
•	Fix: Add a to_device helper at the pipeline edge so the batch inference path can do one pinned-memory copy:
def to_pinned_tensor(vec: np.ndarray) -> torch.Tensor:
    t = torch.from_numpy(vec).pin_memory()
    return t  # caller does .to(device, non_blocking=True)
Then feature_preparer.py should accumulate per-sample vectors and stack once before transfer.
🚀 G-G2 — nx.to_numpy_array(G) materialises an N×N dense matrix for spectral / hybrid embeddings
•	File: src/graph/graph_embeddings.py:229, 239
•	Issue: For long articles entity graphs can have a few hundred nodes; dense float64 adjacency = ~few hundred KB per request, but the eigvalsh peak memory is O(N²) and the call is on the request hot path.
•	Fix: See G-P4 above (sparse Laplacian + sparse eigensolver).

Why not higher
