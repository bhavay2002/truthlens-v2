CRIT E1 — Test set leakage in Evaluator.evaluate (threshold optimization)
File: src/evaluation/evaluator.py:169 174
thresholds = None
if y_proba_arr is not None:
    try:
        thresholds = self.threshold_optimizer.optimize(collected)   # ← fits on the same set being scored
Why it’s critical. collected contains the test labels & probs. We tune the operating point on the test set and report metrics against it. That is textbook test set tuning — every binary/multilabel metric in the report becomes optimistic by 1 5 F1 points (worse on small sets). It also invalidates the multilabel per_label thresholds which were fitted via precision_recall_curve on the same y_true.
Fix. Make threshold fitting an explicit val only step and apply on test:
def evaluate(self, *, val_y_true=None, val_y_proba=None, ...):
    ...
    thresholds = None
    if val_y_proba is not None and val_y_true is not None:
        thresholds = self.threshold_optimizer.optimize({
            "y_true": np.asarray(val_y_true),
            "y_proba": np.asarray(val_y_proba),
            "task": task, "task_type": task_type,
        })
    # apply to TEST predictions
    if thresholds is not None and y_proba_arr is not None:
        y_pred_arr = _apply_thresholds(y_proba_arr, thresholds, task_type)
    metrics = compute_metrics_from_preds(...)
And remove the in-place optimize(collected) call entirely. Add an assert val is not test guard.

CRIT E3 — __aggregate__ averages log_loss together with accuracy/f1
File: src/evaluation/metrics_engine.py:348 363, _aggregate at 442 457.

log_loss ∈ [0, ∞) is on a different scale than the other metrics in [0, 1]. The mean is meaningless and is then exposed to consumers that treat it as a percentage. Worse, EvaluationEngine._extract_val_loss (line 64) reads exactly this aggregated log_loss for early stopping — a single high entropy task swamps everything.
Fix.
•	Drop log_loss from _AGG_KEYS; expose it as agg["log_loss_mean"] only if all tasks are classification.
•	For early stopping, use task weighted mean of per task log_loss with weights = num_samples:
def _aggregate_loss(per_task, weights):
    vals, w = [], []

        if t == "__aggregate__": continue
        ll = (m or {}).get("log_loss")
        if isinstance(ll, (int, float)) and np.isfinite(ll):
            vals.append(ll); w.append(weights.get(t, 1))
    return float(np.average(vals, weights=w)) if vals else None

CRIT E5 — is_binary detection silently misroutes multiclass labelled {0,1}
File: src/evaluation/metrics_engine.py:111 114, mirrored in evaluate_model._infer_task_type:127.

CRIT E7 — Hard-coded 0.5 threshold in 4 places ignores config + per-task tuning
Files: evaluator._postprocess (88 96), evaluate_model._postprocess_logits (108 113), prediction_collector._postprocess (61 65), advanced_analysis.postprocess_predictions:108.

HIGH E1 — Multi-task forward pass is N×slow: shared encoder re-run per task
File: prediction_collector.collect_all_tasks 168 196 and collect_all_tasks_from_loader 232 255 (inner for task in selected_tasks: model(...)).
The TruthLens model has a shared encoder + per-task heads. Running the encoder K times per batch is O(K) inference cost.
Fix. Call once per batch, get all heads. If model returns {"logits": {task: tensor}}:
for batch in dataloader:
    with autocast_context():
        out = model(**inputs)                       # single shared pass
    for task in selected_tasks:
        logits_buf[task].append(out["logits"][task].detach().cpu().numpy())
If multi-head API isn’t supported, add it in src/models/. ~K× speedup on eval (K=5–10 here).
HIGH E2 — Fallback path is per-text Python loop

On 10 k samples this is hours instead of minutes. Always batch — even when predict_batch is missing, mini-batch via predict([t1..tN]) chunks.

HIGH E3 — Repeated np.asarray / torch round-trips in _postprocess
Files: evaluator._postprocess:80 100, prediction_collector._postprocess:42 70, identical pattern in evaluate_model._postprocess_logits.
We np.asarray → torch.tensor → softmax/sigmoid → .numpy() 3× per task. scipy.special.softmax/expit does it in numpy without a tensor allocation:
from scipy.special import softmax, expit
if task_type == "multiclass":
    probs = softmax(arr, axis=-1)
    preds = probs.argmax(axis=1).astype(np.int64)
elif task_type == "binary":
    probs = expit(arr) if arr.ndim == 1 else softmax(arr, axis=-1)[:, 1]
    preds = (probs >= threshold).astype(np.int64)
elif task_type == "multilabel":
    probs = expit(arr); preds = (probs >= threshold).astype(np.int64)
~3× faster, no autograd tensor ever created.
HIGH E4 — extract_confidence definition for binary collapses information
File: calibration.py:61 75, used by compute_calibration:306.
For binary 1D probs = P(class=1), confidence is max(p, 1-p) ∈ [0.5, 1]. But the signed probability is what calibration needs (ECE in expected_calibration_error line 193 198 already uses p[:,1] directly). Mixing the two means the confidence returned in the calibration block (used by task_correlation line 192 and the dashboard) is bimodal/half width — and confidence propagates into task_correlation.compute_task_correlation as global_confidence, where the symmetric mapping kills predictive power.
Fix. Return predicted-class probability for binary in calibration (i.e., p_hat = p if pred==1 else 1-p already equals max(p,1-p) ✓) but also expose prob_positive and feed that into correlation/uncertainty.

Issues:

class VectorTemperatureScaler(nn.Module):

Fairness

def _bootstrap_ci(values, *, n=1000, alpha=0.05):
    rng = np.random.default_rng(0)
    samples = [np.mean(rng.choice(values, size=len(values), replace=True)) for _ in range(n)]
    return float(np.quantile(samples, alpha/2)), float(np.quantile(samples, 1 - alpha/2))
and report only differences whose CIs don’t overlap.
