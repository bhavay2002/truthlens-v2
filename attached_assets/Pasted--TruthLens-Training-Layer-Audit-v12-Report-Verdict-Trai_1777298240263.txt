🛑 TruthLens Training Layer Audit — v12 Report
Verdict: Training cannot run end-to-end as currently wired. Multiple show-stoppers in the wiring layer (create_trainer_fn.py ↔ loss_engine.py ↔ evaluation_engine.py ↔ dataloader_factory.py) prevent a single trainer.train() call from completing on the documented entrypoint (main.py). On top of that, several silent-correctness bugs (AMP grad-norm timing, scheduler defaults, checkpoint state) break stability and reproducibility even when wiring is patched.
________________________________________
🔴 1. CRITICAL TRAINING BUGS
🟥 BUG-1 — build_dataloader signature mismatch (training cannot start)
•	File: src/training/create_trainer_fn.py lines 83–95
•	Function: create_trainer_fn
•	Issue: Calls
build_dataloader(df=train_df, task=task, batch_size=params["batch_size"], shuffle=True)
but src/data_processing/dataloader_factory.py:61 requires:
build_dataloader(*, task, dataset, df, split, config, tokenizer=None)
Missing dataset, split, config. Extra batch_size/shuffle kwargs will raise TypeError.
•	Why it breaks training: First call from main.py → create_trainer_fn → TypeError: build_dataloader() got an unexpected keyword argument 'batch_size'. Training never begins.
•	Exact fix:
from src.data_processing.dataset import TruthLensDataset  # or your factory
from src.data_processing.dataloader_factory import DataLoaderConfig
loader_cfg = DataLoaderConfig(batch_size=params["batch_size"])
train_ds = TruthLensDataset.from_df(train_df, task=task, tokenizer=params["tokenizer"])
val_ds   = TruthLensDataset.from_df(val_df,   task=task, tokenizer=params["tokenizer"])
train_loader = build_dataloader(
    task=task, dataset=train_ds, df=train_df, split="train",
    config=loader_cfg, tokenizer=params["tokenizer"],
)
val_loader = build_dataloader(
    task=task, dataset=val_ds, df=val_df, split="val",
    config=loader_cfg, tokenizer=params["tokenizer"],
)
Add tokenizer to _validate_params required keys.

ttype = cfg.task_type.replace("_", "").lower()  # "multi_class"→"multiclass"

🟥 BUG-4 — Optimizer/scheduler/scaler state never persisted in checkpoints (HARDEN-1 regression)
•	File: src/training/trainer.py lines 197–205, 254–261
•	Function: Trainer._train_epoch, Trainer._save_checkpoint
•	Issue: Calls
self.checkpoint.save(step=self.global_step, model=self._unwrap_model())
with no optimizer=, scheduler=, scaler=. CheckpointEngine.save() writes None for those fields. On resume → AMP loss-scale, optimizer momentum (Adam β₁/β₂ moving averages), and LR-schedule progress are all lost. This silently regresses the HARDEN-1 fix described in replit.md.
•	Why it breaks training: After resume, optimizer restarts from zero momentum → effective LR jumps; GradScaler re-detects the safe scale (~10–100 steps of "calm → spike → calm" pattern); LambdaLR step counter restarts → wrong LR.
•	Exact fix: Pass them through. The optimizer/scheduler/scaler live on TrainingStep so the Trainer needs accessors:
# in Trainer._save_checkpoint and the in-epoch save:
self.checkpoint.save(
    step=self.global_step,
    epoch=self._epoch,                # add an epoch counter
    model=self._unwrap_model(),
    optimizer=self.training_step.optimizer,
    scheduler=self.training_step.scheduler,
    scaler=self.training_step.scaler,
    metrics=metrics,
    tag="epoch",

🟥 BUG-5 — compute_grad_norm runs BEFORE unscale_ under fp16 AMP
•	File: src/training/training_step.py lines 199 vs 211–212
•	Function: TrainingStep.run
•	Issue:
grad_norm = compute_grad_norm(self.model)        # line 199 — gradients still scaled
...
if self.use_amp:
    self.scaler.unscale_(self.optimizer)         # line 212 — too late
Under fp16 AMP, gradients carry the GradScaler loss-scale (typically 2¹⁵–2¹⁶). So the logged grad_norm is ~32 000–65 000× the real value.
•	Why it breaks training: instrumentation.AnomalyClassifier flags grad_norm > 1000 as "exploding_gradients" → fires reduce_lr action every step → TrainingStep halves LR on every step → LR collapses to min_lr within ~30 steps. Spike detector also misfires. Effectively no learning.
•	Exact fix: Move the unscale BEFORE the grad-norm read, and gate the whole block on should_step so accumulation steps don't unscale prematurely:
if should_step:
    if self.use_amp:
        self.scaler.unscale_(self.optimizer)
    grad_norm = compute_grad_norm(self.model)
    if self.config.max_grad_norm:
        torch.nn.utils.clip_grad_norm_(self.model.parameters(), self.config.max_grad_norm)
    ...
else:
    grad_norm = None  # do not log a misleading scaled value
🟥 BUG-6 — _reduce_lr action silently overwritten by scheduler
•	File: src/training/training_step.py:359-362, interaction with src/models/optimization/lr_scheduler.py (LambdaLR)
•	Function: TrainingStep._reduce_lr
•	Issue:
def _reduce_lr(self):
    for g in self.optimizer.param_groups:
        g["lr"] *= 0.5
But the scheduler is a LambdaLR, which on the next scheduler.step() resets g["lr"] = base_lr * lambda(step) — wiping the manual reduction. The intervention is a no-op after one step.
•	Why it breaks training: Spike/anomaly LR control (MonitorAction.REDUCE_LR, TrainAction.REDUCE_LR) appears to fire and log, but has zero effect on the next step's LR. Recovery from a spike never happens.
•	Exact fix: Reduce the LambdaLR's base_lrs (these are what LambdaLR multiplies into) instead of the live lr:
def _reduce_lr(self, factor: float = 0.5):
    for g, base in zip(self.optimizer.param_groups, self.scheduler.base_lrs):
        g["lr"] *= factor
    self.scheduler.base_lrs = [b * factor for b in self.scheduler.base_lrs]
Better: route through the (currently dead) LRSchedulerEngine.adapt(...) — see DEAD-1.
🟥 BUG-7 — build_scheduler defaults num_training_steps=1000; LR decays to 0 mid-training

num_training_steps = len(train_loader) * params.get("epochs", config.training.num_epochs)
scheduler = build_scheduler(optimizer=optimizer, config={
    **params,
    "num_training_steps": num_training_steps,
    "num_warmup_steps": int(0.1 * num_training_steps),
})
Pass that down explicitly; never rely on the silent default.

🟥 BUG-10 — LossEngine double-fires balancer and drops shared_parameters
•	File: src/training/loss_engine.py:113, 130-137 and src/models/loss/multitask_loss.py:236-241
•	Function: LossEngine.compute
•	Issue:

🟠 2. PERFORMANCE BOTTLENECKS
🟧 PERF-1 — Validation tensors detached to CPU per-batch
•	Location: src/training/evaluation_engine.py:193-196
•	Issue: preds.detach().cpu() + labels.detach().cpu() per batch forces device-host sync each batch (full GPU stall). Then streaming metrics use Python-level .sum().item() (another sync per call).
•	Fix: Keep accumulation tensors on GPU; sync only in compute():
class StreamingAccuracy:
    def __init__(self, device): self.correct = torch.zeros((), device=device); self.total = 0
    def update(self, preds, labels):
        self.correct += (preds == labels).sum()
        self.total += labels.numel()
    def compute(self): return (self.correct.item()) / max(self.total, 1)
•	Expected speedup: 3–6× on the validation loop for typical batch counts.
🟧 PERF-2 — Validation runs on every DDP rank without sharding
•	Location: src/training/create_trainer_fn.py:90-95 (no DistributedSampler for val) + Trainer.evaluate
•	Issue: Each rank evaluates the full val set, then _sync_scalar averages identical values. With N GPUs, validation costs N× wall-time vs. proper sharding.
•	Fix: When distributed.initialized, attach DistributedSampler(val_dataset, shuffle=False, drop_last=False) and rely on _sync_scalar to merge per-shard partial sums. For reductions to be correct, switch metrics to (sum, count) reductions, not pre-divided averages.
•	Expected speedup: ~N× for validation on N GPUs.
🟧 PERF-3 — optimize_model calls torch.compile unconditionally on CPU
•	Location: src/training/training_setup.py:208-213
•	Issue: torch.compile adds Dynamo tracing overhead on CPU and silently swallows compile failures via except Exception. On CPU dev/debug runs (the current Replit environment), this slows training and hides real errors.
•	Fix:
if torch.cuda.is_available():
    try:
        model = torch.compile(model, mode="reduce-overhead")
        logger.info("Model compiled with torch.compile")
    except Exception as e:
        logger.warning("torch.compile failed: %s", e)  # don't swallow blindly
•	Expected speedup: Removes a 1.2–2× CPU slowdown on small dev runs; surfaces real compile errors.
🟧 PERF-4 — Wasteful per-micro-batch compute_grad_norm
•	Location: src/training/training_step.py:199
•	Issue: Computes the full grad-norm on every micro-batch even when should_step=False. Each call iterates all parameters and squares norms.
•	Fix: Move into the if should_step: block (also fixes BUG-5).
•	Expected speedup: ~5–10% step latency at gradient_accumulation_steps≥4.
🟧 PERF-5 — _tensor_to_feature_dict flattens float tensors per step
•	Location: src/training/training_step.py:91-104, 128-143
•	Issue: Every 50th step, this iterates all float tensors in the batch, calls .flatten() and float(flat[i]) for up to 50 elements per key — multiple host-device syncs.
•	Fix: Either drop entirely (you already log loss/grad/health) or batch-extract on-device with .detach().cpu().tolist() once per tensor:
for k, v in batch.items():
    if isinstance(v, torch.Tensor) and v.dtype.is_floating_point:
        flat = v.detach().flatten()[:max_items].cpu().tolist()
        feature_dict.update({f"{k}_{i}": x for i, x in enumerate(flat)})
•	Expected speedup: Eliminates ~50 host-device syncs per logging step.

🟡 3. LOSS / OPTIMIZATION ISSUES
🟨 LOSS-1 — Two duplicate SpikeDetector implementations with different policies
•	src/training/instrumentation.py:187-201 (hybrid ratio | z-score)
•	src/training/monitor_engine.py:51-58 (pure ratio)
•	Both fire LR-reduce actions → conflicting interventions per step. Pick one (the instrumentation version is stricter and bias-corrected).
🟨 LOSS-2 — task_scheduler.update_losses only sees one task per step under filtered batching
•	File: src/training/training_step.py:179-182 after _filter_batch (lines 122 + 345-357)
•	The adaptive task scheduler only ever observes the loss of the task it just scheduled → its EMA for unselected tasks stays at the init 1.0 forever → "adaptive" reduces to "round-robin with extra steps".
•	Fix: Run the model on the full batch (no _filter_batch) and let MultiTaskLoss mask via labels — the model is already multi-task, so per-task filtering at the batch level wastes the joint-encoder forward pass and breaks the scheduler signal.
🟨 LOSS-3 — loss = total_loss / grad_accum_steps divides AGAIN after MultiTaskLoss already normalized
•	MultiTaskLoss.forward divides by active_heads (line 254). Then TrainingStep.run divides by gradient_accumulation_steps. Both are correct in isolation, but the per-task gradient magnitude is now 1 / (active_heads × grad_accum), which interacts poorly with the static task_weight (line 209). For grad accum > 1, effective task weights shrink — schedule LR accordingly or scale task_weight by grad_accum.
🟨 LOSS-4 — multilabel_loss ignore mask uses sentinel value -100.0 against float targets
•	src/training/loss_functions.py:55 — multilabel labels are typically {0.0, 1.0}, not -100. The mask targets != -100.0 is always all-True → ignore semantics never trigger. If callers do encode -100, it's compared to a float in safe_targets = torch.where(mask, targets, 0) then ALSO passed through BCE — works, but the contract is undocumented and inconsistent with multiclass_loss's integer ignore.
🟨 LOSS-5 — compute() raises on non-finite per-task loss but MultiTaskLoss.forward already raised
•	src/training/loss_engine.py:122-124 is a redundant double check; not a bug, but the duplicate raise paths confuse stack traces.

🧠 4. MULTI-TASK TRAINING ISSUES
🧠 MT-1 — LossEngineConfig.task_types is built per-task in create_trainer_fn
•	Each Trainer is single-task (task_types={task: task_type}). The "multi-task" loss engine is configured for one task per Trainer instance — so the entire MultiTaskLoss orchestrator (normalizer, coverage, balancer) collapses to single-task. The system's documented "multi-task balanced training" never actually runs.
•	Fix: Either build a single MultiTaskTrainer that shares a backbone across all 6 tasks (that's the original design), or accept that Trainer is per-task and rip out the multitask balancer plumbing here to avoid the false-positive "balanced" claim.
🧠 MT-2 — EvaluationEngine doesn't support binary task type
•	src/training/evaluation_engine.py:138-145 only branches on multiclass | multilabel | regression. Binary is silently dropped (no metric created). Tasks like propaganda (binary in the registry) report no eval score → early stopping never triggers → trains to full epoch budget.
🧠 MT-3 — task_scheduler.next_task() advances state in sanity check
•	src/training/trainer.py:218 → run_sanity_check → training_step.run → task_scheduler.next_task(). After sanity check, the round-robin index is offset by 1 vs. a fresh run. Reproducibility hazard.
•	Fix: Pass a dry_run=True flag through TrainingStep.run that bypasses scheduler/optimizer/scaler mutations during sanity.
🧠 MT-4 — MultiTaskLoss returns task_losses containing weighted, normalized, coverage-multiplied losses, not raw
•	multitask_loss.py:211 task_losses[task] = weighted_loss. The task_scheduler (line 179-182 of training_step.py) consumes these as if they were raw loss magnitudes for adaptive scheduling — they're not. Pass raw_losses instead, OR change the contract.

🚀 5. GPU / DEVICE ISSUES
🚀 GPU-1 — Model is moved to device 3× and re-wrapped under DDP
•	Trainer.__init__: self.model.to(device) (line 75)
•	TrainingStep.__init__: self.model.to(self.device) (line 78) — independently chooses cuda
•	DistributedEngine.wrap_model: model.to(device) again (line 96)
•	The optimizer was built in create_trainer_fn against the original CPU parameters, then the model gets re-moved. PyTorch parameters preserve identity through .to() only when device is the same; cross-device .to() returns a new tensor → the optimizer holds stale refs → "expected all tensors to be on the same device" at first optimizer.step().
•	Fix: Pick one canonical device-move site (in Trainer before build_optimizer), then never move again. Build optimizer after the model is on its final device.
🚀 GPU-2 — non_blocking=True only works with pinned memory
•	training_step.py:333 does v.to(device, non_blocking=True) on every batch. non_blocking is silently ignored unless pin_memory=True AND tensor is pinned. dataloader_factory.py:115 correctly gates pin_memory on CUDA, so this is OK on GPU; on CPU it's a no-op anyway. Check training_utils.move_batch_to_device (line 76) — it correctly gates on is_pinned() — so use that instead of the inline lambda.
🚀 GPU-3 — distributed_engine.initialize() calls torch.cuda.set_device unconditionally
•	distributed_engine.py:78 — when CUDA isn't available (or NCCL isn't) this raises before training starts. Backend should fall back:
if not torch.cuda.is_available():
    self.config.backend = "gloo"
...
if torch.cuda.is_available():
    torch.cuda.set_device(self.local_rank)
🚀 GPU-4 — EvaluationEngine.evaluate does model.to(self.device) per call
•	evaluation_engine.py:113 — moves the model on every eval call. If model is already on device, this is a no-op cost; if it's on a different device (compiled/DDP), it can break. Check before move.
